# Архитектурная схема кластера Proxmox VE MediaHard

Отказоустойчивая трёхточечная схема голосования Paxos/Corosync для ИИ‑платформы MediaHard.

## Общая концепция

Кластер спроектирован по принципу **«Active‑Passive с арбитражем»** для защиты от Split‑Brain и обеспечения непрерывности работы.

**Ключевые механизмы:**
* **Горячее резервирование:** pve2 автоматически подхватывает нагрузку при отказе pve1.
* **Мажоритарный кворум:** 2 из 3 голосов необходимы для работы кластера.
* **Разделение ролей:** вычислительные узлы (pve1/pve2) + арбитр (pve3).

## Детальное описание ролей

### 1. Сервер pve1 (Active Master / Compute Node)

**Железо:** физический мощный сервер (Core i9, 64 GB RAM, All‑Flash NVMe).

**Роль:** основная рабочая нагрузка.
* запускает боевые LXC‑контейнеры (Symfony 7, PostgreSQL 16, Ollama AI);
* служит источником ZFS‑снапшотов для репликации;
* участвует в голосовании Corosync (1 голос).

### 2. Сервер pve2 (Passive Reserve / Replica Host)

**Железо:** физический сервер, симметричный pve1 (Core i9, 64 GB RAM, NVMe).

**Роль:** горячий резерв (High Availability).
* не несёт текущей пользовательской нагрузки;
* принимает ZFS‑снапшоты с pve1 каждые 60 секунд через 10 G‑кабель;
* при аварии pve1 автоматически поднимает контейнеры из свежих снапшотов;
* участвует в голосовании Corosync (1 голос).

### 3. Сервер pve3 (QDevice Witness / Arbiter)

**Железо:** легковесная VM на Hyper‑V (Windows 10) или Raspberry Pi.

**Роль:** арбитр кворума.
* не хранит данные ZFS;
* не запускает код Symfony/Ollama;
* пингует pve1 и pve2 по сети управления (`192.168.2.0/24`);
* отдаёт голос выжившей ноде при разделении сети;
* предотвращает Split‑Brain, обеспечивая мажоритарный кворум (2 из 3);
* участвует в голосовании Corosync (1 голос).
