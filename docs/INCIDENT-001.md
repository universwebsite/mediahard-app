# Инженерный отчет об инциденте #001: Деградация кластера Ceph

**Дата:** 24.04.2024  
**Статус:** Resolved (Решено)  
**Объект:** Инфраструктурный слой MediaHard (Cluster: Proxmox + Ceph)  
**Уровень критичности:** High (Высокий)

## 🚨 Описание ситуации (Symptoms)
В ходе плановой работы кластера была зафиксирована резкая деградация производительности хранилища. 
- На узле `prox3` потребление RAM достигло критического порога.
- Процесс `pvestatd` инициировал «шторм запросов», вызвав лавинообразный рост таймаутов.
- Возник риск потери кворума и развала всей файловой системы CephFS, где хранятся цифровые паспорта товаров.

## 🔍 Диагностика (Analysis)
В ходе анализа логов и мониторинга выявлено:
1. Интенсивный фоновый ребаланс данных (backfilling) потреблял все свободные ресурсы ввода-вывода (IOPS).
2. Наложение фоновой задачи на текущие операции чтения/записи привело к превышению лимитов `osd_memory_target`.
3. Система мониторинга зафиксировала рост `latency` до значений, блокирующих работу PHP-backend приложения.

## 🛠 Предпринятые меры (Action Taken)
Для стабилизации системы были выполнены следующие шаги:
1. **Заморозка ребаланса:** Выполнена команда `ceph osd set nobackfill`. Это позволило немедленно высвободить ресурсы CPU и RAM для обслуживания клиентских запросов.
2. **Тюнинг лимитов:** Оптимизированы параметры кэширования и `osd_memory_target` для предотвращения своппинга на узле `prox3`.
3. **Оптимизация опроса:** Снижена частота опроса статуса нод со стороны `pvestatd` для снижения нагрузки на управляющую сеть.

## ✅ Результат (Outcome)
- Стабильность кластера восстановлена за 15 минут.
- Потери данных (Data Loss) не допущено.
- Доступность маркетплейса сохранена.

## 📚 Выводы и Prevention (Post-Mortem)
Чтобы исключить повторение ситуации, в план работ внесено:
- Настройка Grafana-алертов на порог 85% потребления RAM OSD-демонами.
- Перенос трафика ребаланса Ceph в отдельную физическую сеть (VLAN).
